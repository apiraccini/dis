- **Requirement: VectorStore protocol** — A vector store SHALL expose an async Protocol with operations: upsert, delete_by_document, search.
- **Requirement: Chunk payload** — A chunk record SHALL carry document_id, document_name, tags, chunk_index, and text.
- **Requirement: Sparse vector in chunk payload** — A chunk payload SHALL carry a sparse vector representation (lexical/BM25 term weights) alongside its dense vector when passed to `upsert`.
- **Requirement: Search result** — A search hit SHALL carry document_id, document_name, tags, chunk_index, text, and a similarity score; the score reflects RRF fusion rank rather than raw cosine similarity.
- **Requirement: Filter pushdown** — search SHALL accept optional `tags` and `document_ids` filters applied within the query, returning only hits whose source document matches; tags use OR semantics (document matches if it carries at least one requested tag), document_ids use membership.
- **Requirement: Top-k ranking** — search SHALL return at most `top_k` hits ranked by descending similarity score.
- **Requirement: Atomic upsert** — upsert SHALL replace all existing chunks for a document atomically; re-ingesting the same document overwrites its prior chunks.
- **Requirement: Document name denormalization** — The chunk payload SHALL carry `document_name` so search results return the source filename without a relational join.
- **Requirement: Swappable implementations** — The in-memory implementation SHALL approximate hybrid ranking (RRF fusion over a dense-cosine rank and a lexical-overlap rank) so unit tests exercise the same VectorStore contract without a live Qdrant; a vector-store-backed implementation exists alongside it.
- **Requirement: Backed implementation** — A vector-store-backed implementation SHALL exist behind the VectorStore protocol alongside the in-memory implementation.
- **Requirement: Named vector collection schema** — The vector store SHALL provision the collection with two named vectors per point: `dense` (configured dimension, cosine distance) and `sparse` (no fixed size, native sparse index), created together on first provisioning.
- **Requirement: Collection provisioning** — The vector store SHALL ensure a collection exists on application startup with named vectors `dense` (configured dimension, cosine distance) and `sparse` (native sparse index), creating it only if absent (no destructive recreate on restart); payload indexes SHALL be created on the filtered fields (`document_id`, `tags`) so filtered HNSW traverses filtered-out nodes rather than falling back to full-scan filtering.
- **Requirement: Atomic upsert strategy** — upsert SHALL replace all existing chunks for a document by deleting that document's points first, then inserting the new points; a failure between delete and insert SHALL leave the document's chunk_count unset and surface the error to the ingestion failure path.
- **Requirement: Payload schema** — Each point SHALL store the chunk payload fields (document_id, document_name, tags, chunk_index, text) as point metadata, plus both `dense` and `sparse` named vectors, so search results are populated without a relational join.
- **Requirement: Filter pushdown mechanism** — search SHALL push the tags filter (OR semantics — document matches if it carries at least one requested tag) and the document_ids filter (membership) into the vector query as native filters, not post-filter in application memory.
- **Requirement: Hybrid search fusion** — `search` SHALL query both the `dense` and `sparse` vectors independently (each returning its own top candidates) and fuse the two result sets using Reciprocal Rank Fusion (RRF) into a single ranked list, before applying `top_k` and any tags/document_ids filters.
  - Scenario: keyword-only match ranks higher via fusion — GIVEN a chunk containing an exact keyword match but a mediocre dense similarity, WHEN `search` runs with a query containing that keyword, THEN the chunk ranks higher than it would under dense-only search.
- **Requirement: Delete by document** — delete_by_document SHALL remove all points whose payload document_id matches, and SHALL be a no-op when no such points exist.
